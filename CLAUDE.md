# CLAUDE.md

## Project Overview

**jbeam-edit** is a Haskell CLI tool for parsing, formatting, and editing JBeam files (BeamNG vehicle format). Key features: JBeam parsing with comment preservation, consistent formatting, JBFL rule-based formatting config, and (experimental) node renaming/reference updating.

## Build System

- **Build tools:** Cabal (preferred) or Stack — both are supported. Use a GHC version listed in `tested-with` in `package.yaml`.
- **Source of truth for package config:** `package.yaml` (hpack) — the `.cabal` file is generated from it
- **Do NOT manually edit `jbeam-edit.cabal`** — it is auto-generated by hpack
- **GHC compatibility:** The `base` upper bound in `package.yaml` defines the maximum supported GHC version. Code must build and all tests must pass on the latest GHC within that range, not only the versions listed in `tested-with`.
- **`--prefer-oldest` in CI:** When `tested-with` has more than one GHC version, the CI Cabal build (`cabal.project.ci`) runs with `--prefer-oldest` for the oldest GHC entry, picking the minimum compatible version of every dependency. This does not apply to local builds. Package lower bounds in `package.yaml` must be set high enough to exclude versions that cause build failures under this mode.

### Build commands

```bash
# Build (default)
cabal build

# Build with dev config (tests, LSP, transformation)
cabal build --project-file=cabal.project.dev

# Run tests
cabal test --project-file=cabal.project.dev

# Run the tool
cabal run jbeam-edit -- <file.jbeam>
```

### Cabal flags

Flags are defined in `package.yaml` under `flags:`. Some flags are marked `(experimental)` in their description and gate optional features. If your change touches code in a flag-gated module, enable the corresponding flag when building and testing. All experimental flags are enabled by default in `cabal.project.dev`, so using that project file is sufficient when in doubt. If build times are a concern, you can enable only the relevant flag instead of using `cabal.project.dev` — e.g. `-f transformation` for transformation-only changes.

## Key Architectural Notes

- Parser: Megaparsec-based, preserves comments and whitespace
- Formatter: works on the parsed AST to produce normalized output
- JBFL: mini-language for specifying formatting rules (path selectors + properties like `PadDecimals`, `PadAmount`, `NoComplexNewLine`, `AutoPad`, `Indent`)
- Config files: `~/.config/jbeam-edit/rules.jbfl` (global) or `.jbeam_edit.jbfl` (project-local override)

### NodeCursor / `NC.applyCrumb`

`NodeCursor` tracks the current position in the AST as a sequence of breadcrumbs (`ArrayIndex Int` or `ObjectIndexAndKey Int Text`). `NC.applyCrumb cursor f index node` appends a breadcrumb for `node` at `index` and calls `f updatedCursor node`. Used throughout the formatter to match JBFL rules against the current path.

### FormattingState

Passed through `addDelimiters` / `formatWithCursor` / `doFormatNode`:

| Field                  | Description                                                                                             |
|------------------------|---------------------------------------------------------------------------------------------------------|
| `fsUsePad`             | Whether AutoPad is active                                                                               |
| `fsColumnWidths`       | Max width per column (computed once)                                                                    |
| `fsFormattedCache`     | Pre-rendered cell texts, rows × cols                                                                    |
| `fsHeaderWasExtracted` | Whether the first row was a string header                                                               |
| `fsCurrentRowIdx`      | `Nothing` = outer (row) context; `Just i` = inner (cell) context, `i` indexes into `fsFormattedCache` |

### `Object` children include `Comment` nodes

`Object (Vector Node)` stores both `ObjectKey` and `Comment` nodes as direct children. Any code iterating object contents must handle both constructors. Using an ordered map was considered and rejected specifically because of this requirement.

### `isComplexNode`

A node is complex if it contains an array or object with **more than one child**. Single-child chains (`[[1]]`, `[{a: 1}]`) are **not** complex. `ObjectKey` delegates to its value's complexity. This flag drives all inline vs. multi-line decisions in the formatter — changing its definition affects output broadly.

### Comment association direction

`AssociationDirection` (`PreviousNode` / `NextNode`) is determined during parsing (`Jbeam.hs`), not post-processing. It controls whether a comment renders inline after the preceding node or on its own line before the next one. Changing separator logic in the parser can silently break comment positioning.

### Adding a new JBFL property

`PropertyKey` is a GADT in `Formatting/Rules.hs`. Adding a new property requires updates in all of these places — missing any one causes a compile error:

1. `PropertyKey a` GADT definition
2. `propertyName`
3. `eqKey`
4. `boolProperties` or `intProperties`
5. `parseValueForKey` in `Parsing/DSL.hs`
6. Formatting logic in `Formatting.hs`

### `examples/` directory structure

Used as test fixtures. To regenerate all fixtures after changing formatting logic or example files, run:

```bash
cabal run jbeam-edit-dump-ast --project-file=cabal.project.dev
```

This reformats all files in one pass. You may run this when needed, but **do not suggest committing the results** unless the user has clearly reviewed and approved the fixture changes.

| Directory                      | Contents                               |
|--------------------------------|----------------------------------------|
| `examples/jbeam/`              | Source `.jbeam` files                  |
| `examples/formatted_jbeam/`   | Expected formatter output              |
| `examples/jbfl/`               | JBFL rule configs                      |
| `examples/ast/`                | Haskell AST representations            |
| `examples/invalid_jbeam/`      | Malformed files for parser error tests |
| `examples/transformed_jbeam/` | Expected transformation output         |

## Extracting BeamNG files

To extract JBeam files from BeamNG for testing, always use the extract script:

```bash
bash tools/extract-and-format-jbeam/extract-tune-examples.sh
```

This reads `tools/extract-and-format-jbeam/transform-check-files.txt` for the list of structural files to extract. Do not use `unzip` or manually navigate the BeamNG install directory.
